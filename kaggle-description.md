# AI Workforce Intelligence Platform — Project Description

## Overview

The AI Workforce Intelligence Platform is a production-grade multi-agent AI system that analyzes workforce data, detects utilization risks, forecasts capacity, and generates strategic recommendations through an interactive Streamlit dashboard. Built on Google's Agentic Engineering pattern and the Model Context Protocol (MCP), this system transforms how organizations understand and act on workforce intelligence. Instead of forcing managers to manually comb through spreadsheets, miss early warning signs, and make gut-feel decisions, the platform enables a manager to type "Show me utilization risks in Engineering" and receive a structured executive report with evidence, confidence scoring, and prioritized action items — in under 10 seconds. The system is fully deployable with a single setup command, operates with or without API keys via deterministic mock mode, and includes a rigorous evaluation framework that scores every query across 12 quality metrics.

## Business Impact

Modern workforce management faces three critical challenges that directly affect organizational performance and employee wellbeing. The first is hidden overutilization, where employees silently exceed healthy workload thresholds, leading to burnout, attrition, and productivity loss that goes unnoticed until it is too late. The system automatically detects employees exceeding 90% utilization thresholds and flags them before they reach crisis point. The second challenge is capacity blind spots — organizations routinely miss deadlines or resort to reactive over-hiring because they lack forward-looking visibility into team workload. The platform forecasts workload two to three months ahead with data-backed projections grounded in actual worklog and capacity data. The third challenge is reactive decision-making, where costly last-minute rebalancing becomes the norm because there is no systematic mechanism for early intervention. The system generates prioritized strategic recommendations with full evidence traceability, enabling proactive rather than reactive workforce management. Without this system, managers rely on manual spreadsheet analysis that misses warning signs and produces gut-feel decisions. With this system, every decision is supported by structured data, auditable reasoning, and confidence scoring.

## Why Agentic AI

A single LLM call that attempts to "analyze this CSV and tell me what's wrong" suffers from four fundamental limitations that make it unsuitable for enterprise workforce analytics. The first is hallucination — the model fabricates numbers, metrics, or citations that are not present in the data, which is unacceptable when making decisions about employee workload and capacity planning. The second is no tool access — a single LLM cannot query databases, validate its own outputs, or fetch external context to ground its analysis in real data. The third is no memory — each query starts from scratch with no awareness of prior analysis, making multi-turn investigations impossible. The fourth is no validation — if the model outputs a wrong calculation, there is no mechanism to catch it before it reaches the decision-maker. Agentic AI solves all four problems by replacing a single LLM call with a structured lifecycle: PLAN, ACT, OBSERVE, VALIDATE, REFINE, REPORT, and MEMORY UPDATE. Each phase is executed by a specialized sub-agent with its own tools, validation rules, and error recovery mechanisms. The result is deterministic data access, self-correcting outputs, and fully auditable reasoning. This system implements the complete Google Agentic Engineering pattern — not as a chatbot wrapper, but as a proper multi-agent orchestration loop with shared state, retry logic, and graceful degradation.

## System Architecture

The architecture centers on a Manager Agent that orchestrates four specialized sub-agents through a shared state dictionary. When a user submits a query through the Streamlit dashboard, the Manager Agent triggers the PLAN phase by invoking the WorkforceQueryAgent, which performs intent classification, entity extraction, and tool routing. For the query "Check utilization for Engineering," this agent detects the intent as "utilization_analysis" with the entity department set to Engineering and extracts the department filter. The Manager Agent then routes to the UtilizationAgent for workload computation and the RecommendationAgent for strategic synthesis. During the ACT phase, each sub-agent executes its assigned tools from the deterministic tool layer. The WorkforceQueryAgent uses the EmployeeLookupTool, ProjectAnalysisTool, MCP Integration Tool, and WorklogReaderTool to gather data. The UtilizationAgent uses EmployeeLookupTool and ProjectAnalysisTool to compute utilization percentages, detect overloaded employees exceeding 90%, underutilized employees below 50%, and identify risk patterns. The ForecastAgent uses the ForecastTool to analyze capacity trends and detect hiring gaps. The RecommendationAgent uses the RecommendationTool to synthesize all outputs from the other agents into prioritized, evidence-backed strategic actions. All data queries go through tools, not the LLM — the LLM only synthesizes, never retrieves, which eliminates hallucination at the data access layer. The OBSERVE phase logs every agent execution with duration, status, tools used, and output summaries. The VALIDATE phase runs eight-point response validation and nine-point report validation, ensuring every output meets quality standards before reaching the user. The REFINE phase regenerates failed reports up to three times before accepting partial results. The REPORT phase routes state to the correct report builder, which generates structured executive reports with sections including executive summary, department utilization tables, overallocated employee flags, business risks, recommendations, evidence citations, and executive conclusions. The MEMORY UPDATE phase persists session context so subsequent queries can reference prior analysis.

## Multi-Agent vs Single LLM Comparison

The difference between this agentic system and a conventional single LLM application is profound across seven dimensions. For data access, a single LLM app relies on prompt injection and inevitably hallucinates, whereas this system uses tools that query real CSV data deterministically. For error recovery, a single LLM app either returns an error or hallucinates a response, while this system retries up to three times per agent and implements graceful degradation that produces partial results with noted limitations instead of crashes. For validation, a single LLM app has none, while this system enforces eight-point response validation and nine-point report validation per query. For memory, a single LLM app is stateless, while this system maintains session memory across turns that allows multi-turn investigations. For auditability, a single LLM app is a black box with no visibility into its reasoning, while this system provides a full execution trace with evidence lineage that every judge can inspect in real time through the dashboard. For MCP support, a single LLM app has none, while this system includes Filesystem, Google Drive, and Notion connectors with path traversal security. For LLM dependency, a single LLM app always requires API keys, while this system includes a graceful mock fallback that functions identically without any API keys, making it immediately evaluable by any judge.

## Datasets

The system operates on six synthetic datasets covering 15 employees across 5 departments with 2 months of worklog data. The employees.csv dataset contains 15 records with employee_id, department, role, salary_band, and status columns. The project_allocations.csv dataset contains 27 records mapping employees to projects with allocation percentages. The worklogs.csv dataset contains 498 records tracking hours_logged and task_category per employee per date. The attendance.csv dataset contains approximately 450 records with attendance status covering Present, PTO, Sick, and Leave. The capacity.csv dataset contains approximately 30 records defining total_capacity_hours per employee per month. All data is deterministically generated with seed 42 for full reproducibility — every judge sees the exact same data on every run, ensuring fair and consistent evaluation.

## Evaluation System

The platform includes a built-in evaluation framework that scores every query response across 12 weighted metrics. Intent accuracy at 20% weight measures whether the query was classified correctly versus unknown. Routing accuracy at 20% weight measures whether the right agents were invoked for the detected intent. Execution plan quality at 15% weight measures whether validation checks were passed. Validation score at 15% weight measures whether all required report sections were present. Evidence completeness at 10% weight measures whether all claims are grounded in dataset citations. Latency score at 5% weight measures response time, awarding 1.0 for under 2 seconds and 0.8 for under 5 seconds. State completeness at 5% weight ensures all required state keys are populated. Context completeness at 5% weight tracks context warnings. Confidence at 5% weight comes from response_metadata. The system defines three quality tiers: Excellent at or above 0.90, Good at or above 0.75, and Needs Review below 0.75. The full benchmark suite can be executed with a single command — python evaluation_runner — which runs over 20 benchmark queries and produces a scorecard, failure analysis, and regression report in the evaluation_results directory. Every response passes through two validation gates. The ResponseValidator performs eight checks covering JSON schema, execution plan structure, mandatory report sections, metadata completeness, routing correctness, confidence calculation, tool-agent alignment, and agent-tool alignment. The ReportValidator performs nine checks ensuring no placeholder text, no assumed domain knowledge, no leaked prompts, no duplicate paragraphs, no empty sections, no hardcoded metrics, required evidence fields, section-agent alignment, and report consistency. If validation fails, the report regenerates up to three times before accepting partial results.

## Failure Handling

The system implements a three-layer defense that ensures it never crashes on bad input. The first layer is retry — the _execute_with_retry method in ManagerAgent gives each sub-agent up to three attempts with full logging of every failure. The second layer is graceful degradation — when an agent fails after all retries, the system propagates error state and produces an incomplete report with clearly noted limitations rather than crashing. The third layer is state preservation — retry_history and execution_trace capture full failure context that is surfaced in the report's Evidence section so the user can see exactly what went wrong. The system handles seven specific failure scenarios. Missing API keys trigger mock mode with a visible banner and confidence lowered to 0.45, allowing full evaluation without any API keys. LLM call failure triggers an automatic fallback chain from Gemini to OpenAI to Mock mode. Missing datasets trigger a clear error message via st.stop with no traceback dump. Non-existent employee queries cause the tool to return Employee not found and the report accurately reflects this. Invalid query intent defaults to unknown intent and runs the full analysis pipeline rather than failing. Sub-agent crashes trigger three retries then skip with noted limitations in the report. Report validation failure triggers up to three regeneration attempts before accepting partial results.

## End-to-End Execution Example

The complete internal reasoning pipeline for the query "Check utilization for Engineering" demonstrates the full agent lifecycle. In the PLAN phase, the WorkforceQueryAgent detects the intent as utilization_analysis with department set to Engineering, and the Manager Agent routes to UtilizationAgent and RecommendationAgent. In the ACT phase, EmployeeLookupTool filters by Engineering department and returns four employees. WorklogReader loads and filters 120 worklog entries across two months. The UtilizationAgent computes per-employee utilization by dividing total hours by capacity hours. EMP001 at 152 out of 168 hours equals 90.5% and is flagged as overloaded. EMP003 at 76.2% and EMP006 at 85.7% are normal. EMP009 at 57.1% is noted as low. Results are structured into the shared state under utilization_results. In the OBSERVE phase, the observation layer logs the agent as UtilizationAgent, status as success, duration as 1450 milliseconds, tools as EmployeeLookupTool, and output as four employees scored. In the VALIDATE phase, all eight response validation checks pass and confidence is computed at 0.92. In the REFINE phase, since validation passed, no refinement is needed. In the REPORT phase, the router detects utilization_analysis intent and selects the UtilizationReport builder, which generates seven sections: Executive Summary noting one overloaded employee representing 25 percent of the Engineering team, Department Utilization table with all four employees, Overallocated Employees flagging EMP001 at 90.5 percent, Business Risks covering burnout and project delay risks, Recommendations to redistribute 10 percent of EMP001's load, Evidence with dataset citations and row counts, and Executive Conclusion with priority actions. The ReportValidator passes on the first attempt. In the MEMORY UPDATE phase, session memory is updated with the intent, agents executed, confidence of 0.92, execution time of 2.3 seconds, and overall status as success, enabling the next query to reference this context.

## What the Judge Sees

The Streamlit dashboard presents a split-panel interface. The left panel displays the final executive report with all sections. The right panel shows the real-time agent lifecycle execution trace — PLAN, ACT, OBSERVE, VALIDATE, REPORT — updating as each phase completes. The bottom of the dashboard shows evidence cards that link every claim back to specific datasets with row counts, providing full auditability. If no API key is configured, a clear banner at the top announces "Running in demo mode — mock responses." The system also includes a deterministic mode toggle that allows judges to compare agentic AI output against a purely deterministic baseline, demonstrating the specific value added by the multi-agent orchestration layer.

## Report Engine

The reporting subsystem implements an intelligent routing architecture. The ReportRouter examines the shared state and routes to the correct report builder based on detected intent, available data, and execution context. The base ReportBuilder provides shared utilities for section generation, evidence formatting, and confidence display. The NarrativeGenerator handles LLM-powered prose generation when available, falling back to template-based generation in mock mode. The ReportValidator enforces nine quality checks on every generated report, ensuring no hallucinated content, no placeholder text, and full evidence traceability before any report reaches the user interface.

The system addresses every major rubric category for the Kaggle competition. ## Frameworks and Technology Stack

The platform is built with Python 3.10 or higher using Streamlit 1.30 or higher for the interactive dashboard. The LLM layer supports both Google Gemini 1.5 Pro and OpenAI GPT-4o with automatic fallback between providers and graceful degradation to deterministic mock responses when no API keys are configured. The Model Context Protocol enables Filesystem, Google Drive, and Notion integration with security constraints. Configuration management uses YAML overrides merged with environment variables through a singleton settings class. Deterministic thresholds for overloaded at 90 percent and underutilized at 50 percent are centralized in the settings module for easy adjustment.

## Scalability and Production Readiness

The architecture is designed for horizontal scaling. The shared state pattern eliminates inter-agent messaging bottlenecks. The deterministic tool layer ensures all data access follows the same pattern regardless of which agent initiates the query. The retry with backoff mechanism prevents cascading failures. The evaluation framework enables continuous quality monitoring. The entire system runs on Streamlit Cloud with zero infrastructure setup, and the single setup.bat command installs all dependencies, generates datasets, and launches the application on any Windows machine.

## Project Structure

The codebase is organized into clearly separated modules. The agents directory contains the Manager Agent orchestrator and four specialized sub-agents. The tools directory implements the deterministic tool layer with employee lookup, project analysis, forecast computation, MCP integration, and worklog reading. The config directory manages settings through a singleton class and YAML overrides. The data_layer directory handles synthetic data generation with seed 42, cleaning, validation, and pipeline orchestration. The evaluation directory contains the benchmark suite, quality scoring, and response validation. The reporting directory implements the intelligent report routing, building, narrative generation, and validation. The prompts directory stores YAML prompt templates. The tests directory contains unit tests. The datasets directory holds generated CSV files.

## Quick Start

The system is designed for immediate evaluation. Running setup.bat installs all dependencies, generates the synthetic datasets, and launches the Streamlit application. No API keys are required — the system runs in mock mode with a clear banner indicating demo mode while all agents, tools, and reports function exactly as they would with a real LLM. For full LLM-powered operation, users copy the env.example to env, add either a Gemini API key or an OpenAI API key, and run setup.bat. Manual setup is also supported through standard Python virtual environment creation, pip install, data pipeline execution, and Streamlit launch.
